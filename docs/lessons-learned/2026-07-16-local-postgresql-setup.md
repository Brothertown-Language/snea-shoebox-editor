# Local PostgreSQL Setup — Development Instance Guide

Date: 2026-07-16

## Context

The project uses a local PostgreSQL instance for development and regression testing. The local database is a production replica managed via a pgserver instance at `tmp/local_db/` and synced from production using `scripts/sync_prod_to_local.py`. This document covers setup, management, and common interaction patterns.

## pgserver Location

- **pgserver data directory:** `tmp/local_db/` (relative to project root)
- **Configuration file:** `tmp/local_db/postgresql.conf` — auto-generated by pgserver
- **pgserver binary:** Installed as a project dependency via `pyproject.toml`; runs a self-contained PostgreSQL instance

The pgserver instance binds to `localhost` on the default PostgreSQL port (5432). It starts automatically when needed by scripts and notebooks that call `get_session()`.

## Sync Mechanism

The local database is synced from production via `scripts/sync_prod_to_local.py`. This script:

1. Drops all local tables
2. Recreates them by introspecting the production `pg_catalog` schema (exact column types, constraints, indexes, generated columns)
3. Copies production data into the recreated tables using parameterized INSERT batches (1,000 rows per batch)
4. Resets auto-increment sequences to match the max ID in each table

For the complete technical deep-dive on the sync mechanism, see `docs/lessons-learned/2026-06-14-pg-catalog-schema-replication.md`.

### Sync Command

The synchronous shell wrapper is located at `scripts/sync_prod_to_local.sh`:

```bash
bash scripts/sync_prod_to_local.sh
```

This wrapper must be run before every regression test cycle per the Regression Test Protocol in `AGENTS.md`.

### Companion Script

For heavy schema resets where external `pg_dump`/`psql` binaries are acceptable, use `scripts/reset_schema_to_prod.py`. This is a separate path from the primary pg_catalog-based sync.

## Starting and Stopping

The pgserver instance manages its own lifecycle. In normal development workflows:

- **Start:** pgserver starts on demand when the first database connection is opened via `get_session()`. No manual start command is needed in most cases.
- **Stop:** pgserver stops when the owning process exits. To stop a background instance, use `pkill pgserver` or stop the Jupyter kernel that holds the connection.
- **Status:** Check `tmp/local_db/` for the presence of a running postmaster.pid file, or attempt a lightweight connection via psycopg2.

## Query Patterns

### Using SQLAlchemy (Preferred)

The project uses SQLAlchemy for all database access. The primary entry point is `get_session()`:

```python
from src.database.connection import get_session

with get_session() as session:
    records = session.execute("SELECT * FROM records LIMIT 5").fetchall()
    for row in records:
        print(row.id, row.mdf_data)
```

### Direct psycopg2

For ad-hoc queries outside the ORM:

```python
import psycopg2

conn = psycopg2.connect("postgresql://postgres:@localhost:5432/postgres")
cur = conn.cursor()
cur.execute("SELECT count(*) FROM records")
print(cur.fetchone())
```

### Example: Searching `mdf_data` Columns

The `records` table has an `mdf_data` column containing linguistic data. To search for entries matching a pattern:

```python
with get_session() as session:
    results = session.execute(
        "SELECT id, mdf_data FROM records WHERE mdf_data ILIKE :pattern",
        {"pattern": "%\\ph%"}
    ).fetchall()
    print(f"Found {len(results)} entries")
```

This query returns approximately 430 entries matching the `\ph` phonological pattern.

## Recommendation

Use `get_session()` for all programmatic database access. Use `bash scripts/sync_prod_to_local.sh` to refresh the local database before running regression tests. Prefer the pgserver-managed instance over manual PostgreSQL installations to ensure schema compatibility.
